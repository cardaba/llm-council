<objective>
Cablear el **routing por profile end-to-end para `fast` y `quality`** en backend, sin todavía implementar `quality_research` (eso es Plan 03-04). Resultado tras este plan: el usuario puede enviar `profile: "fast"` o `profile: "quality"` desde un cliente HTTP (curl/Postman) y obtener una deliberación con el council mix correspondiente. `profile: "quality_research"` será aceptado por Pydantic pero el código ejecutará un placeholder explícito (`raise NotImplementedError` o fallback a `quality`) hasta que Plan 03-04 lo conecte al strategy module.

Purpose: Romper el problema en dos pasos para minimizar el blast radius. Antes de tocar la complejidad del research_strategy, validar que el contrato HTTP + el branching por profile en council funciona limpiamente con los dos profiles "lineales".

Output:
- `SendMessageRequest` con `profile: Literal["fast", "quality", "quality_research"] = "fast"` (D-28 + D-29).
- Endpoints `/message` y `/message/stream` propagan `request.profile` a `run_full_council`.
- `run_full_council(user_query, profile)` toma el profile, lee `PROFILES[profile]`, y enruta:
  - `fast` y `quality`: usa `council_models` y `chairman_model` del profile en lugar de los aliases globales.
  - `quality_research`: placeholder (`raise NotImplementedError("quality_research lands in Plan 03-04")` — Plan 03-04 lo reemplaza con el delegate al strategy module).
- Las funciones `stage1_collect_responses`, `stage2_collect_rankings`, `stage3_synthesize_final` aceptan ahora `council_models` / `chairman_model` como argumento explícito (en vez de leer del global) — refactor mecánico que preserva su comportamiento.
</objective>

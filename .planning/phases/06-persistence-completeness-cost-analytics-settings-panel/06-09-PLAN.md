<objective>
Close the gap exposed by manual smoke-testing of Plan 06-08: the frontend guard correctly prevents the crash, but the underlying deliberation is silently aborted by uvicorn when the SSE client disconnects (`asyncio.CancelledError` propagates through the `async for event in research_strategy.run(...)` block, gets caught by the generic `except Exception` at backend/main.py:370-372, emits an `error` event to the now-dead connection, and exits without calling `storage.add_assistant_message`). The user's deliberation â€” and the API spend that paid for it â€” is lost.

Verified diagnosis (2026-05-11): a curl `--max-time 3` abort against `/api/conversations/{id}/message/stream` with `profile=fast` leaves the persisted JSON at 389 bytes with only the user message, title still "New Conversation", no assistant entry.

Output: `backend/main.py` refactored so each SSE handler's deliberation runs as a detached `asyncio.create_task(...)` fed through an `asyncio.Queue`. The HTTP-bound `event_generator` only drains the queue and yields SSE strings â€” it never owns the deliberation. Client disconnect cancels the generator but leaves the deliberation task running; the task completes, persists, exits cleanly.

This is "Option B" from the gap analysis discussion. Options A (try/finally guard around partial state) and D (incremental persistence) were rejected because they only preserve partial deliberation â€” Option B preserves the complete result and matches the user's expectation ("no pierdo el trabajo cuando cambio de conversaciĂłn").
</objective>

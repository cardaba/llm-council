<objective>
Close the BLOCKER race condition uncovered during 06-UAT Test 2: when the user switches conversations via `handleSelectConversation` while an SSE stream from the previous conversation is still in flight, the in-flight events keep firing `setCurrentConversation((prev) => ...)`. The setter's `prev.messages` now belongs to the NEW conversation (loaded from JSON via `loadConversation` / `api.getConversation`). Persisted messages do NOT carry a `loading` object — only freshly-sent in-flight messages do. So `lastMsg.loading.stage1 = false` throws `TypeError: Cannot set properties of undefined (setting 'stage1')`, the App component unmounts, and the user must manually reload.

Purpose: surgical defensive guard. This is Option 1 from the UAT gap entry (the "guard the read" approach) — the minimal-surface patch that closes the BLOCKER without changing the streaming UX. Options 2 (AbortController cancellation in `handleSelectConversation`) and 3 (per-event `conversation_id` scoping with backend changes) are explicitly deferred to v2.1+ as proper hardening — they're better fixes but introduce trade-offs (Option 2 kills the original stream on switch; Option 3 requires backend SSE shape changes). For v2.0 gap closure we ship the smallest patch that unblocks the user.

Output: `frontend/src/App.jsx` with an `if (!lastMsg?.loading) return prev;` early-return inserted at the top of every `setCurrentConversation((prev) => ...)` setter inside `handleStreamEvent` that writes to `lastMsg.loading.*` or to in-flight assistant slots (`stage1` / `stage2` / `stage3` / `stage4` / `metadata` / `critic`).

Classification note (from UAT): this race condition is **pre-existing**, not a regression introduced by Phase 6. Phase 6 added `settingsOpen`, `costStatsRefreshTrigger`, `useSettings()`, and `stage4Threshold` — none of which touch `handleStreamEvent`. The vulnerability has existed since SSE streaming was introduced. The plan is filed under Phase 6 because UAT surfaced it.
</objective>

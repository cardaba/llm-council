<objective>
Complete the SET-03 vertical: the slider value flows from `useSettings().stage4Threshold` (localStorage), through `ChatInterface` (which gates on profile === 'quality_research'), through `api.sendMessageStream` (which adds `stage4_threshold` to the JSON body), through `SendMessageRequest.stage4_threshold` Pydantic validation (`ge=1, le=10`), into `research_strategy.run(..., threshold_override=...)`, and finally consumed at the critic gating step in place of the static PROFILES default.

Purpose: Without this plan, the slider is decorative — it persists to localStorage but never reaches the backend, so the user's "stricter / looser" choice has no effect on Stage 4 refinement. SET-03 closes the loop.

Critical constraint: BACKWARD COMPATIBILITY. v1 requests (no `stage4_threshold` field) MUST continue to validate and run with the static PROFILES default. Pitfall 3 (RESEARCH.md lines 438-447) is the canonical failure mode — `None < threshold` raising TypeError because the fallback wasn't applied.

Output: A measurable end-to-end loop where moving the slider in the Settings panel changes the threshold value sent on the NEXT quality_research request (and only on quality_research — fast/quality requests omit the field).
</objective>

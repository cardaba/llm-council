    1. **Frontmatter / encabezado:**
       - Título: "Cognitive Walkthrough — LLM Council UI Baseline (Phase 2)".
       - Fecha de captura, comit/branch del baseline auditado.
       - Actor profile: usuario único, perfil BI/data, single-user local app, sin onboarding previo.

    2. **Sección "## Method"** explicando que cada paso de cada flujo se evalúa con las 4 preguntas Nielsen-Polson:
       - Q1: ¿Qué intenta el usuario en este paso?
       - Q2: ¿Está la acción correcta disponible?
       - Q3: ¿Es la acción correcta visible / descubrible?
       - Q4: ¿El feedback resultante es comprensible y conecta con la intención?
       Y que cada friction point se etiqueta con `Friction F-XX` (numeración global) con: descripción, evidencia (componente.jsx:linea o copy literal observado), severity hint (low/med/high — el Plan 02 lo re-puntúa con Nielsen 0–4), e implicación para redesign.

    3. **Cuatro secciones de flujo, en este orden y con estos headings exactos** (necesario para grep):
       - `## Flow 1: Cold start` — primer arranque, sin conversaciones. Debe cubrir al menos: ¿cómo descubre el usuario que tiene que crear una conversación?, ¿qué le dice la app sobre qué es?, ¿el welcome state empuja al CTA o se queda mudo?, ¿qué pasa si el sidebar está vacío?
       - `## Flow 2: Ask + review` — escribir pregunta → SSE streaming → tabs Stage 1 → Stage 2 (de-anonimización + aggregate rankings) → Stage 3 → download markdown. Debe cubrir al menos: ¿el usuario sabe que la respuesta tarda 15-30s?, ¿el progreso entre stages es visible?, ¿la diferencia entre Stage 1/2/3 es legible?, ¿el usuario entiende que los modelos están evaluándose anónimamente?, ¿el download produce el formato esperado?
       - `## Flow 3: Manage conversations` — three-dot menu hover, modal de delete con confirmación, inline rename con commit-on-Enter/blur y cancel-on-Escape, search progresivo (title-first + affordance "Search inside content (N)" con threshold ≥3 chars). Debe cubrir al menos: ¿el three-dot es descubrible si el usuario no hace hover?, ¿el modal comunica destructividad?, ¿el inline rename comunica que está editable?, ¿el affordance de content-search es entendible o suena a feature oculta?, ¿qué pasa con la conversación activa si la borras?
       - `## Flow 4: Attachments` — drag-and-drop + file picker, chip list, eliminación de attachment, errores de tamaño (500KB/file, 2MB total). Debe cubrir al menos: ¿es descubrible que se puede adjuntar?, ¿los chips comunican qué tipo de archivo es?, ¿el error de tamaño aparece dónde y cómo?, ¿qué le pasa al texto del prompt cuando se adjunta?

    4. **Para cada flujo: subsecciones `### Step N: <accion concreta>`** (numeradas dentro del flujo) con las 4 preguntas Q1-Q4 respondidas, y un bloque `**Friction F-XX:**` cuando la respuesta a alguna Q expone un problema.

    5. **Sección final `## Friction Index`** — tabla con todos los `F-XX` listados: ID, flujo, paso, descripción de una línea, evidencia, severity hint. Esta tabla es la que Plan 02 (Nielsen audit) y Plan 03 (redesign) van a consumir.

    Restricciones obligatorias:
    - **Concreto, no genérico.** Cada friction point debe citar el componente y línea aproximada (`Sidebar.jsx:198` o similar) o el copy literal observado en pantalla (entrecomillado). No vale "el usuario podría confundirse" sin evidencia.
    - **Mínimo 12 friction points totales** repartidos entre los 4 flujos (al menos 2 por flujo). El UI tiene Bootstrap-flavored defaults sin mucho cuidado UX, así que encontrar friction es fácil — el riesgo es subajustar.
    - **No proponer redesign aquí.** Cada friction tiene un campo `Implication` que apunta dirección genérica ("falta affordance de hover", "falta progress communication") pero NO propone solución concreta — eso es Plan 03.
    - **El walkthrough cubre lo que YA existe** — Phase 1 está en master, Phase 3/4 todavía no. Por tanto el Quality toggle, el reasoning_details disclosure y el branded header NO entran en este walkthrough (eso lo cubren los anticipatory findings del Plan 02 Nielsen audit). Sí entran las 4 features Phase 1: modal de delete, three-dot menu, inline rename, search progresivo.
    - **Idioma:** español para la prosa, identificadores y headings principales en inglés (Flow 1, Step 1, Friction F-01, Q1..Q4) para que el grep sea estable.

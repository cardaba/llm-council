# Feature Research — v2.0 Milestone

**Domain:** Personal multi-LLM deliberation app, single-user, local-only. Adding 7 features on top of shipped v1.0.
**Researched:** 2026-05-10
**Confidence:** MEDIUM-HIGH (UX patterns drawn from observed peers — ChatGPT Custom GPTs, Cursor cost trackers, Linear/Obsidian/Notion settings panels, Playwright community guidance. WebSearch verified for cost-ticker and Playwright; pattern claims for Custom GPT / Claude.ai Projects rely on documented behavior rather than fresh interaction tests.)

> **Scope discipline.** This file ONLY covers the 7 v2.0 features (A-G in MILESTONES). Existing v1.0 surface (Quality dial, Stage 1/2/3, anonymized peer review, BYOK, SSE per-stage, file attachments text-only, conversation CRUD, ErrorBanner, theme toggle, Direction A skin, reasoning_details disclosure) is taken as given and NOT re-researched.
> **Tone constraint.** All UI patterns must respect Direction A "Research notebook" — calmo, no alarmista, editorial, whitespace generoso, microcopy de 12px en `--color-fg-muted` para metadata. Anti-features below explicitly call out anything that violates this.

## Feature Landscape

### Table Stakes (Users Expect These)

Features that, if missing, make the v2.0 milestone feel half-shipped or buggy.

| # | Feature | Why Expected | Complexity | Notes |
|---|---------|--------------|------------|-------|
| A1 | **Critique mode entry point clearly distinct from fresh-prompt** | If both flows live in the same screen, users will accidentally trigger the wrong one. Custom GPTs solve this with a dedicated "Knowledge" tab; Claude.ai Projects with a dedicated workspace; Cursor with `@` mention. The user must immediately read "I'm in critique mode" without reading docs. | MEDIUM | Recommended pattern: **segmented control on the welcome state** (`Ask council` / `Critique research`) — same surface, two entry modes. Shows BOTH options up-front so user discovers it without setup. Avoid hidden modal or "Advanced" submenu. |
| A2 | **3 distinct upload slots, one per Quality-mode council model, with model attribution visible** | The whole point is "this file was generated by ChatGPT o3 / Claude Opus / Gemini Pro and the council critiques each". Generic "drop files here" loses the attribution. ChatGPT's knowledge base shows file names but no per-source attribution because it doesn't matter for them; for us it does. | MEDIUM | Pattern: **3 labeled drop zones stacked vertically**, each pre-labeled with the model name (`OpenAI o3 research`, `Anthropic Opus research`, `Gemini Pro research`) — derived from `COUNCIL_MODELS` in `config.py`. Each slot accepts ONE file, replacing on re-drop. File icon + filename + size + remove (×) chip after upload. |
| A3 | **Inline preview of uploaded content (collapsible)** | Users won't blindly trust a 50KB file went in correctly. Standard pattern: collapsed accordion showing first ~10 lines + "Show full" expansion. Cursor `@file` shows full content inline; ChatGPT shows nothing (problematic — F-26 anti-pattern from v1 walkthrough). | MEDIUM | Render markdown (reuse `<Markdown>` component) inside an accordion (reuse the `grid-template-rows: 0fr → 1fr` CSS-only technique from `ReasoningDisclosure`). Truncate at 500 lines visible; show "Showing 500 of 1247 lines" footnote. |
| A4 | **File size + format limits surfaced before upload** | F-26..F-28 from v1.0 walkthrough already documented this gap. Critique mode files will be 5–50KB realistically; cap at 200KB/file × 3 files = 600KB total to leave headroom under existing 2MB attachment ceiling. | LOW | Microcopy below the 3 slots: `.md or .txt · up to 200KB per file`. Reuse `formatBytes` helper from `download.js`. |
| A5 | **Critique persists like normal conversations** | If the conversation list shows critique sessions identically to fresh-prompt sessions but reload loses the uploaded files, that's a P0 bug. Sidebar must distinguish: small icon (📄 or `crit:` prefix) signaling mode. | MEDIUM | Persist `mode: "critique" \| "fresh"` + `critique_files: [{model_label, filename, content, size}]` in conversation JSON. Sidebar item gets a mode indicator (subtle icon or pill, NOT loud). |
| B1 | **Stage 2 metadata (label_to_model, aggregate_rankings) survives reload** | Currently a known v1.0 bug (CONCERNS.md, PERS-V2-01): Stage 2 evaluations look orphaned when reloading historical conversations because the de-anonymization map is gone. Users assume the system remembers what they saw 5 minutes ago. | LOW | Schema migration: add `metadata: {label_to_model, aggregate_rankings}` to assistant message persistence. Backward-compat: if absent, render Stage 2 in legacy mode (current behavior) without breaking. |
| C1 | **Per-message cost shown after deliberation** | If you advertise a Quality dial with cost surfacing, users expect to see actual cost post-hoc, not just predicted cost pre-hoc. Cursor's status bar shows `$0.04 this chat`; OpenRouter API returns usage in response — easy to capture. | LOW-MEDIUM | OpenRouter responses include `usage: {prompt_tokens, completion_tokens, total_cost}`. Capture in `query_model`, sum per stage, store in `metadata.cost`, render as microcopy footnote in `MessageHeader` (e.g., `Quality+Research · $0.34`). |
| C2 | **Per-session cumulative cost** | Users want to know "how much did today's queries cost me?" without parsing OpenRouter dashboard. Standard placement: sidebar footer (Linear, Cursor) — discreet, always visible, never shouts. | LOW | Sum `metadata.cost` over today's conversations. Render in sidebar footer as `$1.42 today · 7 queries`. Reset at midnight local time (no DB needed; just filter by `created_at` date). |
| D1 | **Settings page reachable without breaking existing layout** | If we move the theme toggle into settings AND remove it from the header, users will hunt for it. Alternative: leave a quick toggle in header AND deeper controls in settings. Pattern: gear icon → opens panel/modal. | LOW | Recommended: **slide-out panel from right**, 380px width, semi-transparent overlay. Triggered by gear icon in header (next to current theme toggle). Theme toggle stays in header for quick access; settings panel duplicates the control alongside other prefs. |
| D2 | **stage4_threshold tunable** | v1.0 known debt: hardcoded `stage4_threshold = 8` may exhibit chairman self-preference bias. Without UI tuning, users cannot calibrate. | LOW | Slider 1-10 with current value displayed. Microcopy: `Higher = stricter (Stage 4 refinement only when critic confidence is high)`. Persist in `localStorage` (single-user, no backend needed). |
| D3 | **Font-size override + density preference** | Listed as v2.0 backlog. Direction A locks 17px body Source Serif 4; users with 4K monitors or vision needs deserve a knob. | LOW | Radio S/M/L (15px / 17px / 19px) + density compact/comfortable (line-height 1.55 / 1.65). Apply via CSS variable override on `:root`, persist localStorage. |
| E1 | **Touch targets ≥44×44 throughout** | Apple HIG and Material baseline; below 44×44 is a usability bug on mobile, not an enhancement. Audit existing buttons (close-modal ×, sidebar three-dot menu, theme toggle, attachment chip remove). | LOW | CSS audit pass + min-height/min-width rules. No new components — adjust existing CSS. |
| E2 | **Sidebar drawer accessible via swipe gesture from left edge** | iOS/Android native pattern; users expect it. Without swipe, users must reach for the 44×44 toggle button — usable but feels primitive. | MEDIUM | Plain `touchstart/touchmove/touchend` on a 20px-wide invisible edge zone; threshold ~50% drawer width. No library needed. |
| E3 | **Focus trap when drawer/modal open** | Accessibility table stake (WCAG 2.4.3). Tab key must not escape an open drawer/modal into the page underneath. | LOW | `<Drawer>` / `<Modal>` wrapper with `inert` attribute on siblings (modern browsers 2024+) OR focus-trap-react (3KB) if `inert` polyfill needed. |
| E4 | **Viewport meta + safe-area-insets for notched devices** | Without it, content goes under notch / home indicator on iPhone. Trivial fix that makes the app feel mobile-native instead of "desktop site shrunk". | LOW | `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">` + `padding: env(safe-area-inset-top) env(safe-area-inset-right) env(safe-area-inset-bottom) env(safe-area-inset-left)` on shell. |
| F1 | **Snapshot tests for key surfaces (welcome, Stage 3, ErrorBanner, sidebar empty, light+dark)** | The point of visual regression testing is to lock the design after months of investment. Without it, the next CSS refactor silently regresses Direction A. | MEDIUM | Playwright `toHaveScreenshot()` on 5 surfaces × 2 themes = 10 baselines. CI runs in Docker (Linux) to avoid Windows anti-aliasing drift. |
| F2 | **Wait-for-fonts before snapshot** | woff2 self-hosted fonts (Source Serif 4, Inter, JetBrains Mono variable) load asynchronously; without `document.fonts.ready` await, snapshots intermittently capture system-ui fallback and fail. | LOW | `await page.evaluate(() => document.fonts.ready)` before each `toHaveScreenshot()`. |
| F3 | **Threshold tuning for cross-platform stability** | Default `threshold: 0.2, maxDiffPixelRatio: 0.01` is community baseline; pixel-perfect (`threshold: 0`) is brittle on Linux+Docker. Lock to one OS in CI to dodge the worst drift. | LOW | Run snapshots in Linux Docker only (CI). Local devs on Windows update baselines via `--update-snapshots` and reviewer checks Linux CI run before merge. |
| G1 | **Backend test for storage UUID validation boundary cases** | SEC-01 fix in v1.0 (Vuln 2). If a refactor breaks this regex, the path traversal vuln is back. Tests prevent silent regression. | LOW | pytest async client: invalid UUIDs (`../`, empty, wrong format, valid UUID for non-existent file) → 400 / 404. ~6 test cases. |
| G2 | **Backend test for council profile routing** | The Quality dial only works if `Fast`/`Quality`/`Quality+Research` route to different model sets. A typo in `PROFILES` would silently degrade to wrong model. | MEDIUM | pytest: mock `query_model` to return canned responses, assert `query_model.call_args_list` matches expected models per profile. Each profile = 1 test. |
| G3 | **Backend test for research_strategy critic parser malformed responses** | The critic parser is the most fragile path: malformed score string → conservative skip. If parser breaks, Stage 4 fires when it shouldn't (cost!) or never (quality regression). | MEDIUM | pytest with 5-7 malformed input fixtures (missing score, score > 10, non-numeric, empty, unicode garbage) → conservative-skip behavior asserted. |
| G4 | **Frontend test for useTheme hook (matchMedia + localStorage)** | Theme is the most visible global state; bug here is immediately noticed. Test `prefers-color-scheme` change, localStorage persistence, manual toggle. | MEDIUM | vitest + @testing-library/react + jsdom matchMedia mock. ~4 test cases. |
| G5 | **Frontend test for QualityToggle state propagation** | If toggle stops propagating profile to API request, all queries silently default to Fast or Quality. Highest blast radius for least code. | LOW | vitest: mount, click each segment, assert `onChange` callback fires with correct profile string. |

### Anti-Features (Commonly Requested, Often Problematic)

Features that look reasonable but violate Direction A tone, scope, or single-user discipline. Document explicitly so they don't quietly creep back in.

| # | Anti-Feature | Why Requested | Why Problematic | Alternative |
|---|--------------|---------------|-----------------|-------------|
| AA1 | **LLM-powered settings explainer ("Ask AI to set up your preferences")** | Trendy in 2025–2026; ChatGPT, Notion AI, Linear all flirted with it. Feels "modern". | Complexity creep at zero benefit. Settings are 4 controls; a tooltip suffices. Adds an LLM call cost to a free action. | Plain microcopy under each control (`Higher = stricter` for stage4_threshold). Zero LLM calls in the settings page. |
| AA2 | **Real-time animated cost ticker counting up during deliberation** | Cursor Pulse does this; visually engaging. | Direction A is "calmo, no alarmista". A counter racing upward turns a thinking tool into a slot machine. Triggers anxiety, not transparency. | Show final cost AFTER stage 3 completes, as static microcopy in `MessageHeader`. Cumulative session cost in sidebar footer updates on conversation save (not in real-time). |
| AA3 | **Daily/weekly cost alerts with email or push** | "I spent $50 yesterday and didn't realize" pain. | Single-user local app — there is no email infra, no service worker for push. The OpenRouter dashboard already does this. Adding it is rebuilding what exists. | Sidebar footer showing today's spend is enough peripheral awareness. If a hard cap is needed, set it on OpenRouter side, not here. |
| AA4 | **In-app pricing comparison table for council models** | Power users like dashboards. | Out-of-date the moment it ships (provider price changes monthly). Maintenance burden with zero compounding value. | Tooltip with `~$X/1k tokens (BYOK rate)` on the model chip — verified at milestone start, accept staleness risk. Link to provider docs for ground truth. |
| AA5 | **Drag-to-reorder file slots in critique mode** | Notion-y feature, satisfying. | The slot ↔ model mapping is semantically fixed (file from Opus must go in the Opus slot). Reordering breaks the mental model. | Slots stay in `COUNCIL_MODELS` order, labeled with the model name. User drops the right file in the right slot, period. |
| AA6 | **Auto-detect which model generated each uploaded research** | "Smart upload" feel. ChatGPT recently shipped it for some flows. | Brittle (writing styles converge), invasive (would need to send file content to a classifier LLM = extra cost), wrong attribution silently breaks Stage 2 anonymization. | User explicitly drops the file in the right slot. Trust the user. |
| AA7 | **Settings as full-page route via React Router** | "Real" SPA pattern. | Requires adding `react-router-dom` (~10KB), changing nav model, and creating a back-button paradigm in a single-shot conversation app. Disproportionate. | Slide-out panel from right (D1). Closes with Escape or click-outside. |
| AA8 | **Mobile UI parity — every desktop feature available on phone** | "Mobile first" dogma. | The app is for serious thinking; you're not realistically reading 2000 words of Stage 2 deliberation on a phone. Optimize for sidebar-on-mobile + read final answer; tolerate horizontal scroll on Stage 2 raw evaluations. | Mobile gets: drawer sidebar, full-screen Stage 3, readable Stage 1 tabs. Stage 2 raw evaluation tabs may scroll horizontally — acceptable. |
| AA9 | **Dark mode auto-switch by time of day (sunset trigger)** | Apple Reading Mode does it. | The user already has OS-level prefers-color-scheme + manual toggle in v1.0. Adding a third trigger just causes "why did the theme change?" confusion. | Keep current useTheme behavior. |
| AA10 | **100% test coverage** | Looks rigorous. | Tests have a maintenance cost; chasing coverage numbers leads to brittle tests of trivial code. The MILESTONES.md target of ~60% on critical paths is honest. | Test the 5-7 paths in G1-G5 above. Skip getters/setters and dumb prop pass-through. |
| AA11 | **Visual regression on every component, not just key surfaces** | More tests = more safety, naively. | Snapshot count × OS count × theme count = exponential. After ~30 snapshots maintenance dominates. Lock the 10 highest-value surfaces and accept that smaller components are caught by component tests. | F1 lists 5 surfaces × 2 themes = 10. Stop there for v2.0. |
| AA12 | **Streaming token-by-token in Stage 3 chairman synthesis** | Looks "modern". | Current architecture is per-stage SSE events. Adding token-streaming is a new event type, frontend rerender per token, and backend buffer changes. Out of scope. | Per-stage SSE remains; chairman thinking time is communicated by the existing Stage 1 progress strip. |
| AA13 | **Onboarding tour / guided tooltips for v2.0 features** | "First-run UX". | Single user. They built the app. They know the features. | Tooltips on hover for non-obvious controls (stage4_threshold microcopy). No coachmarks, no skippable tour. |

## Feature Dependencies

```
A. External Critique
    ├──requires──> A5 persist mode + critique_files
    │                  └──depends-on──> B. label_to_model + aggregate_rankings persistence schema
    │                                      (same conversation JSON migration)
    └──reuses──> existing Stage 1/2/3 council pipeline
                     └──existing v1.0 surface, no change needed

C. Cost Analytics
    ├──requires──> capture OpenRouter `usage` field in query_model (backend hook)
    └──C2 sidebar cumulative ──depends-on──> C1 per-message cost (sums per-message values)

D. Settings Page
    ├──D2 stage4_threshold tuning ──ENABLES──> v1.0 known-debt resolution
    ├──D3 font-size + density ──independent──> CSS variable override
    └──no external dependency; pure localStorage + UI

E. Mobile responsive
    ├──E1 touch targets ──foundation──> no dependency
    ├──E2 swipe gestures ──depends-on──> existing W23 drawer (v1.0)
    ├──E3 focus trap ──depends-on──> Drawer/Modal component refactor (or wrapper)
    └──E4 safe-area-insets ──independent──> CSS only

F. Visual regression
    ├──depends-on──> Direction A locked (DONE in v1.0)
    ├──depends-on──> woff2 fonts loading deterministically (DONE in v1.0)
    └──F1 snapshots ──complement──> G frontend tests (different layers, both needed)

G. Automated tests
    ├──G1 storage UUID ──depends-on──> v1.0 SEC-01 fix (DONE)
    ├──G2 profile routing ──depends-on──> v1.0 PROFILES dict (DONE)
    ├──G3 critic parser ──depends-on──> v1.0 research_strategy.py (DONE)
    └──G4-G5 frontend ──independent──> vitest setup is new infra

CRITICAL CHAIN:
    B (persistence schema migration)
    ↓
    A5 (critique persistence reuses same migration)
    ↓
    C1 (cost added to same metadata block)

Doing B first opens a clean schema migration that A5 and C1 piggyback on. Doing them in
random order = three separate migrations + three reload-compatibility shims.
```

### Dependency Notes

- **B precedes A5 and C1 (schema migration ordering):** All three add fields to the assistant message persistence model. Doing B first means one migration, one `metadata` block; doing them separately means three. Roadmap should land B before or in the same phase as A5.
- **D (Settings) is a dependency for D2 (stage4_threshold tuning):** Without a settings surface, there is nowhere to put the slider. If D ships first, D2 follows trivially. If D2 ships standalone (e.g., env var only), it's invisible to users — partial value.
- **E1 (touch targets) is a foundation for E2/E3:** A swipe gesture on a 32×32 button is usability theater. Audit and fix targets before wiring swipe.
- **F (visual regression) and G (automated tests) are complementary, not redundant:** F catches CSS regressions invisible to G; G catches logic regressions invisible to F. Both needed; neither replaces the other.
- **A (Critique) and Quality+Research mode coexistence:** The Quality+Research flow uses `research_strategy.py`; critique mode uses the basic council pipeline. They must not interfere — confirm that profile routing in `main.py` is mode-aware (`if mode == "critique": use COUNCIL_MODELS, ignore research strategy`).

## MVP Definition

This is a SUBSEQUENT milestone (v2.0). MVP framing here = "what must ship in v2.0 for the milestone to be coherent" vs "what can slip to v2.1 without breaking the story".

### Launch With (v2.0)

The seven features as scoped in MILESTONES.md. Within each, the table-stakes subset:

- [ ] **A1-A5** External Critique entry point: segmented control + 3 labeled slots + collapsible preview + size limits surfaced + persistence — *the new core feature; without all 5, the mode feels broken*
- [ ] **B1** Stage 2 metadata persistence — *fixes a known v1.0 bug; trivial relative to value*
- [ ] **C1, C2** Per-message cost + sidebar cumulative — *the cost story is part of the Quality dial value prop; v2.0 closes it*
- [ ] **D1, D2, D3** Settings panel + stage4_threshold + font-size/density — *a settings page with one knob is half-shipped; ship 3*
- [ ] **E1, E3, E4** Mobile touch targets + focus trap + safe-area-insets — *accessibility + notched-device baseline. E2 swipe is differentiator, not table stake*
- [ ] **F1, F2, F3** Visual regression snapshots, font-ready wait, threshold tuning — *the lock-the-design value*
- [ ] **G1-G5** Critical-path test coverage backend + frontend — *60% target per MILESTONES*

### Add After Validation (v2.x)

Defer until v2.0 has been used for 2-4 weeks and gaps surface:

- [ ] **A6** Side-by-side critique comparison view — *trigger: user says "I can't tell which critique attacked which file"*
- [ ] **A7** Diff highlighting between research and critique — *trigger: user manually copy-pastes recommendations into source*
- [ ] **C3** Cost breakdown by stage — *trigger: user asks "why is Stage 2 so expensive?"*
- [ ] **C4** Cost-per-1k-tokens tooltip — *trigger: user starts comparing models*
- [ ] **D4** Data folder path display + Open in Finder — *trigger: user backs up data manually and gets the path wrong*
- [ ] **D5** Export all conversations as zip — *trigger: machine migration*
- [ ] **E2** Sidebar swipe gesture — *trigger: mobile use frequency increases*
- [ ] **F4** Hover/focus state visual regression — *trigger: a microinteraction regression slips through*
- [ ] **G6, G7** Accordion grid-trick test, E2E critique flow test — *trigger: a regression in one of these areas*

### Future Consideration (v3+)

Genuinely defer; not on the immediate radar:

- [ ] Multi-turn within a critique (follow-up question on the chairman's recommendation) — *out of scope per PROJECT.md "single-shot" decision*
- [ ] Custom critic rubric per critique session — *adds complexity to the rubric system, low ROI*
- [ ] Comparing two critique sessions side-by-side — *narrow use case*

How peer tools handle the v2.0-relevant patterns. Used as reference for "what does the genre look like" not as a copy template.

| Pattern | ChatGPT Custom GPTs | Claude.ai Projects | Cursor (IDE) | Continue.dev | Linear / Notion / Obsidian | Our Approach |

## Key Patterns to Adopt vs Avoid

### Adopt (and adapt to Direction A)

1. **Cursor's status-bar cost display** → re-skin as static microcopy in MessageHeader + sidebar footer. Same idea, less anxious presentation.
2. **Custom GPT 3-slot knowledge layout** → adapt to per-model labeled slots. Reuse the "drop or click to upload" affordance.
3. **Linear's slide-out preferences panel** → exactly the pattern for D1 settings. Right-edge panel, click-outside or Esc closes.
4. **Obsidian's accordion sources** → exactly the pattern for A3 file preview (we already use this technique in ReasoningDisclosure).
5. **Playwright community baseline** (`threshold: 0.2, maxDiffPixelRatio: 0.01`, Linux Docker, font-ready wait) → adopt verbatim.

### Avoid

1. **ChatGPT's no-attribution file list** — for our use case attribution IS the feature (A2).
2. **Cursor's racing cost ticker** — violates Direction A "calmo" tone (AA2).
3. **Notion's full-route settings** — disproportionate; settings panel is enough (AA7).
4. **GPT-builder's onboarding wizard** — single-user app doesn't need it (AA13).

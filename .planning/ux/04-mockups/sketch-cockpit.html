Should we use FastAPI's BackgroundTasks or Celery for our async workload (50 jobs/min, mostly DB queries)?

Stage 1 · Individual responses 4 / 4 OK

For 50 jobs/min with mostly DB queries, FastAPI BackgroundTasks works only if you can afford to lose pending work on restart. Celery is the safer choice when you need persistence, retries, or a separate scaling axis for workers vs. web.

Decision pivots: (a) durability across restarts, (b) jobs occasionally exceed the request lifetime, (c) team comfort running Redis or RabbitMQ as broker.

Reasoning · 1,247 tokens · 2.3s · gpt-5.1
Considered three deployment shapes: single-node uvicorn, multi-node behind nginx, and ECS Fargate with autoscaling. BackgroundTasks loses pending jobs on container replacement; this is the dominant failure mode at 50 jobs/min. Celery+Redis adds ~80MB memory baseline but unlocks retries, scheduled tasks, and observability. RQ is a lighter alternative if Redis is the only broker. Final ranking weighted durability and operational maturity above raw throughput.

Stage 3 · Synthesis FINAL · CHAIRMAN: gemini-2.5-pro

Use Celery with Redis as the broker. At 50 jobs/min with DB-bound work, BackgroundTasks will silently drop pending jobs on container restart — the failure mode you'll regret. Celery gives you durability, retries, and a separate scaling axis with modest operational overhead.

If you want a lighter footprint and won't need scheduled tasks, RQ is a reasonable middle path — but Celery is the answer the council converges on.

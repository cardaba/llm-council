You asked
Should we use FastAPI's BackgroundTasks or Celery for our async workload (50 jobs/min, mostly DB queries)?

Stage 1

Four models answered in parallel.

For 50 jobs/min with mostly DB queries, FastAPI BackgroundTasks works only if losing pending work on restart is acceptable. Celery is the safer choice once you need durability, retries, or a separate scaling axis for workers vs. web.

The decision pivots on three questions: durability across restarts, whether jobs can exceed the request lifetime, and how comfortable the team is running a broker like Redis or RabbitMQ.

Considered three deployment shapes — single-node uvicorn, multi-node behind nginx, and ECS Fargate with autoscaling. BackgroundTasks loses pending jobs on container replacement; this is the dominant failure mode at 50 jobs/min. Celery+Redis adds modest baseline memory but unlocks retries and observability. RQ is a lighter middle option if Redis is already in the stack.

Final answer

Use Celery with Redis as the broker. At 50 jobs/min with DB-bound work, BackgroundTasks will silently drop pending jobs every time a container restarts — the failure mode you'll regret. Celery gives you durability, retries, and a separate scaling axis with modest operational overhead.

If you want a lighter footprint and won't need scheduled tasks, RQ is a reasonable middle path — but the council converges on Celery.

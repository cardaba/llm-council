## v2.0 Requirements

### Critique mode (CRIT) — feature A

**Goal:** Parallel entry point that lets the user upload externally-generated deep research files (one per Quality-mode council model) and have the council critique them via the existing 3-stage flow.
**Outcome:** Shipped end-to-end in Phase 5 (Plans 05-02 backend + 05-03 frontend + 05-04 reload hydration). `POST /critique/stream` accepts multipart, `stage1_collect_responses` parametrised with `external_context`, anonymisation strips model names, ExternalResearchPanel hydrates on reload.

- [x] **CRIT-01**: A new entry point lets the user choose between "Fresh question" and "Critique research" before starting a deliberation. The fresh-prompt flow (textarea + Quality dial) is preserved bit-for-bit when "Fresh question" is selected. — **Validated** (Phase 5 Plan 03 CritiqueWelcome + segmented control).
- [x] **CRIT-02**: 3 file-pickers per Quality-mode council model, `.md`/`.txt` only, 750KB cap Pydantic-enforced. — **Validated** (Phase 5 Plan 02 + Plan 03 DropZoneSlot).
- [x] **CRIT-03**: `POST /api/conversations/{id}/critique/stream` accepts multipart/form-data, emits the existing SSE event protocol. — **Validated** (Phase 5 Plan 02 endpoint, byte-identical SSE shape).
- [x] **CRIT-04**: Stage 1 produces 3 critiques where each council member sees ALL three deep researches with authorship labels visible. Implementation reuses `stage1_collect_responses` with `external_context: Dict[model_id, {filename, content}]`; no new isolated module. — **Validated** (Phase 5 Plan 02 council.py extension).
- [x] **CRIT-05**: Stage 2 anonymised peer-review strips explicit model names before concatenation. — **Validated** (Phase 5 Plan 02 anonymisation pass).
- [x] **CRIT-06**: Pre-flight token estimate blocks runs >150K tokens; Stage 2 truncates each critique to first 600 tokens with `[…truncated]` marker. — **Validated** (Phase 5 Plan 02).
- [x] **CRIT-07**: Pre-flight cost estimate in UI, soft rate-limit in localStorage (5 critiques/hour). — **Validated** (Phase 5 Plan 03 + rate-limit util).
- [x] **CRIT-08**: Reloading a critique conversation hydrates files as collapsed chips via `grid-template-rows: 0fr → 1fr` accordion. — **Validated** (Phase 5 Plan 04 ExternalResearchPanel).

### In-conversation navigation (NAV) — feature H (added by user post-research)

**Goal:** Long deliberations stay oriented through sticky headers + nav strip + show-more + back-to-top.
**Outcome:** Shipped in Phase 5 Plan 05. **Note:** post-execution manual smoke (2026-05-11 milestone close) surfaced a positioning bug — sticky containers don't share scroll-container top edge, producing a "ghost strip" of scrolling code above the sticky bar. Letter of NAV-01 satisfied (`position: sticky` applied) but spirit needs revisit. Tracked as P0 regression in `.planning/v2.1-BACKLOG-FROM-MANUAL-SMOKE.md`.

- [x] **NAV-01**: Stage section headers sticky to top of main panel (under global Header at 52px). — **Partial** (position: sticky applied; visual layering needs v2.1 fix per manual smoke).
- [x] **NAV-02**: Horizontal StageNavigationStrip with chips per stage, scroll-spy highlight via IntersectionObserver. — **Validated** (Phase 5 Plan 05).
- [x] **NAV-03**: Long Stage 1 responses (>600px) collapse to Show-more accordion; historical conversations default to collapsed on first render. — **Validated** (Phase 5 Plan 05).
- [x] **NAV-04**: Back-to-top floating button after >800px scroll, honours `prefers-reduced-motion`. — **Validated** structurally (Phase 5 Plan 05). Polish gaps (contrast, `aria-label`) noted in v2.1 backlog P2.

### Persistence completeness (PERS) — feature B

**Goal:** Persist all metadata produced during deliberation including Stage 2 mappings, so historical conversations hydrate fully on reload.
**Outcome:** Phase 5 Plan 01 (schema migration) + Phase 6 Plan 02 (label_to_model/aggregate_rankings packing) shipped together. `metadata` dict is now opaque-contract: extends without `add_assistant_message` signature changes.

- [x] **PERS-01**: `label_to_model` + `aggregate_rankings` persisted inside `metadata` dict on every assistant message; both fast/quality and quality_research pack them. — **Validated** (Phase 6 Plan 02).
- [x] **PERS-02**: Reloading a v2.0 conversation hydrates Stage 2 metadata fully — no "Quality (legacy)" fallback. — **Validated** (Phase 6 Plan 02 + Phase 7 Plan 04 MessageHeader test).
- [x] **PERS-03**: v2.0 writes carry `schema_version: 2`; v1.0 conversations lazy-migrated server-side in `get_conversation` via `_migrate_conversation_if_needed`. — **Validated** (Phase 5 Plan 01, Phase 7 Plan 04 migration roundtrip test passes).

### Cost analytics (COST) — feature C

**Goal:** Capture, aggregate, and surface real OpenRouter spend per profile per session; show BYOK upstream cost (not covered by $100/month cap).
**Outcome:** Phase 6 Plans 03 + 04. **Note:** v2.1 backlog flags the cost footer placement as "visually disconnected" — Phase 6 D-02 dual-column lock may need revisit.

- [x] **COST-01**: `query_model` captures `usage.cost` + `upstream_inference_cost`; propagates into `metadata.cost = {stage1..stage4, total, upstream_total, currency}`. — **Validated** (Phase 6 Plan 03 + 06-01 spike confirmed field shape).
- [x] **COST-02**: `MessageHeader` displays static cost microcopy line below profile/models metadata. No animated tickers. — **Validated** (Phase 6 Plan 04 + Phase 7 Plan 05 vitest).
- [x] **COST-03**: `GET /api/stats/cost` returns current_month aggregate + cap remaining. Read-only from `data/conversations/*.json` filtered by `created_at`. — **Validated** (Phase 6 Plan 04 `backend/stats.py`).
- [x] **COST-04**: Sidebar footer displays cumulative cost + queries + cap %. Progress bar at ≥80%. — **Validated** (Phase 6 Plan 04). Polish gap (visual disconnection) tracked in v2.1 backlog P2.

**Outcome:** Phase 6 Plans 05–07. Pattern (`<dialog>` + `showModal()` + localStorage individual-key) reused for Phase 7 mobile drawer with zero rework.

- [x] **SET-01**: Gear icon in `Header.jsx` opens 380px slide-out panel; pure `useState` boolean, no React Router. — **Validated** (Phase 6 Plans 05 + 06).
- [x] **SET-02**: 4 controls (theme, font-size S/M/L, density compact/comfortable, stage4_threshold 1-10); `useSettings()` hook (~30 LOC, mirrors `useTheme`). — **Validated** (Phase 6 Plan 06 + Phase 7 Plan 05 vitest).
- [x] **SET-03**: `stage4_threshold` travels in request body as `Optional[int] = Field(None, ge=1, le=10)`; backend falls back to PROFILES default; v1 requests unchanged. — **Validated** (Phase 6 Plan 07).
- [x] **SET-04**: Density applies via FOUC blocker (synchronous before first paint). Font-size via React state (acceptable single-render flicker). — **Validated** (Phase 6 Plan 05 + 06).

### Mobile responsive (MOBL) — feature E

**Goal:** Fully usable ≤768px (touch targets, focus trap, safe-area).
**Outcome:** Phase 7 Plans 01 + 02. MOBL-04 scope-cut candidate was resolved as **NO CUT** (D-01 in 07-CONTEXT) — full swipe gesture shipped with edge-zone + vertical-dominance gates.

- [x] **MOBL-01**: Touch targets ≥44×44px enforced via `--touch-target-min` CSS variable consumed in component CSS. — **Validated** (Phase 7 Plan 01).
- [x] **MOBL-02**: Mobile sidebar drawer uses native `<dialog>` + `showModal()` + `inert` on main content. NO library, NO hand-rolled trap. — **Validated** (Phase 7 Plan 02 SidebarDrawer.jsx, mirror of SettingsPanel.jsx pattern from Phase 6).
- [x] **MOBL-03**: `viewport-fit=cover` meta + `env(safe-area-inset-*)` for header/drawer/composer padding. — **Validated** (Phase 7 Plan 01 with defensive extension to all 4 sides on composer).
- [x] **MOBL-04**: `useTouchSwipe` hook (~30 LOC) opens drawer on left-edge swipe-right, closes on swipe-left; `touch-action: pan-y` on `.messages-container`. — **Validated** (Phase 7 Plan 02; final LOC = 34, both gates present).

### Visual regression testing (VRT) — feature F

**Goal:** Lock the Direction A skin via Playwright snapshot tests.
**Outcome:** Phase 7 Plan 03. **Two adjustments mid-flight:**
- **D-02 override:** scope expanded from 5×2=10 baselines (ROADMAP default) to 8×2=16 (added Settings + Stage 2 + Critique surfaces) per user CONTEXT decision.
- **D-03 revision:** Docker-only requirement was relaxed to native Windows execution. Cross-OS pixel-perfect guarantee is theoretical for a single-user no-CI project; Docker remains as v2.1+ contingency.

- [x] **VRT-01**: `frontend/visual-tests/` contains Playwright specs. **Override:** 8 surfaces (welcome, Stage 3, ErrorBanner, sidebar empty, theme toggle, Settings, Stage 2, Critique) × 2 themes = 16 baselines (vs ROADMAP 5×2=10). — **Validated** (Phase 7 Plan 03, 16 PNGs committed, second-run validation clean).
- [x] **VRT-02**: 5 anti-flake measures applied: `document.fonts.ready`, `threshold: 0.2`, `maxDiffPixelRatio: 0.02`, `reducedMotion: 'reduce'`, animation override via `page.addStyleTag`. Centralised in `_fixtures.ts` for structural enforcement (D-02c). — **Validated** (Phase 7 Plan 03).
- [x] **VRT-03**: Baselines exclusively from a single OS. **Revised:** native Windows (not Linux Docker) given single-user no-CI posture. Docker tag `mcr.microsoft.com/playwright:v1.59.1-noble` remains as v2.1+ contingency if CI is later introduced. — **Validated** (Phase 7 Plan 03 + revised CONTEXT D-03).
